# StatsCompute Performance Deep-Dive

Ревью после применения первого раунда фиксов. Текущие измерения (1080p
preview, Xiaomi Mi Pad 1 / Tegra K1):

```
demosaic = 9.17 ms  (stable)
stats    = 20–23 ms (avg ~22)
blit     = 1.45 ms
GPU tot  = 33 ms
FPS      = ~22
```

Demosaic, Blit, RgbaToNv12 — закрыты. Основной остаточный тяжеловес —
StatsCompute.

---

## Что съедает 22 мс в StatsCompute — пересмотренная модель

Первая оценка этого документа (divmod в loop + арифметика как главные
паразиты) была **неверна**. Две предложенные микрооптимизации регрессировали
на железе — см. "Что не сработало" ниже. Актуальная модель ниже.

### Реальные компоненты стоимости

| Работа | Оценка | Комментарий |
|---|---|---|
| 9 `texelFetch` × 2M пикс | ~8–12 мс | L1 tex-cache на Tegra K1 очень маленький (≤12 КБ), с 6 резидентными WG тычется о потолок |
| 9 `dot(rgb, LUMA_COEF)` × 2M пикс | ~3–5 мс | 3 FMAD каждый |
| `atomicAdd(sHist[w][bin])` × 2M | ~2–4 мс | shared-atomic, intra-warp serialize при flat-luma |
| Reduction (barriers + tree) | ~1 мс | |
| 32K global `atomicAdd` финал | ~1–2 мс | L2 contention |
| Divmod в loop | ~1 мс *(не 4!)* | компилятор видит `patchW` как uniform-divisor → magic-number reciprocal |
| **Итого** | **~16–25 мс** | |

**Главный паразит — L1 tex cache pressure**, а не арифметика. Каждый WG
держит в L1 ~3–4 строки изображения, 6 WGs резидентных → ~18–24 строк
одновременно. Tegra K1 L1 tex cache физически этого не вмещает — идёт
частичный eviction и повторное заполнение.

## Что не сработало и почему

### ❌ Warp-aligned 2D iteration (−13 мс регрессии)

Идея была заменить flat-stride `for (p = tid; p < patchPx; p += 256)` на
warp-выравненную развёртку: warp 0 идёт по строкам 0, 8, 16, …; warp 1 по
1, 9, 17, … и т.д.

**Почему на бумаге казалось выигрышем:** уходят divmod, внутри warp'а `gy`
uniform → coalesced row fetch.

**Почему на железе регрессировало:**

- Оригинальный flat-stride держит working set компактным: все 8 warp'ов
  WG маршируют по соседним 2–3 строкам, Sobel расширяет до 3–4 строк на WG.
  → L1 footprint на SMX: 6 WG × 3–4 строк ≈ 18–24 строк.
- Warp-aligned растягивает warp'ы на 8 разных строк **одновременно**. Sobel
  расширяет до ~10 строк на WG. → L1 footprint: 6 WG × 10 строк ≈ 60 строк.
- Tegra K1 L1 tex cache физически не вмещает 60 строк 1920×4 → **L1 thrash**.

Плюс стоимость divmod была переоценена: компилятор видит `patchW` как
uniform-divisor и генерирует magic-number reciprocal (2–3 cycles), а не
эмуляцию деления (30 cycles). "Экономия 4 мс" была воображаемой.

**Урок:** на Tegra K1 с одним SMX и маленьким L1, **warp-шаблон, где warp'ы
идут соседними строками, лучше warp-шаблона, где warp'ы раскинуты по
диапазону строк**. Компактность working set > warp-local uniform py.

### ❌ Sobel subsample по `py & 3 == 0` (в паре с warp-aligned: ещё хуже)

При warp-aligned итерации `py` детерминирован для каждого warp'а:

- Warp 0: py ∈ {0, 8, 16, …} → все ≡ 0 mod 4 → **100 % Sobel**.
- Warp 4: py ∈ {4, 12, 20, …} → все ≡ 0 mod 4 → **100 % Sobel**.
- Warps 1, 2, 3, 5, 6, 7: `py & 3 ≠ 0` всегда → **0 % Sobel**.

Два warp'а тянут весь Sobel, остальные простаивают на барьерах редукции.
Warp-skew → barrier ждёт самого медленного. Регрессия усилилась.

**Урок:** любая "каждая N-я" подвыборка должна распределяться равномерно по
warp'ам. Только пространственный шаблон по `py` вместе с warp-aligned
iteration даёт перманентный skew.

## Новый план: ранжирование по риску и эффекту

| Путь | Ожидание | Риск регрессии | Сложность |
|---|---|---|---|
| **Temporal subsample (stats каждый 2-й кадр)** | ~11 мс среднее | **Нулевой** (host-логика, GPU не трогаем) | 30 минут |
| Luma scratch image | ~3–5 мс | Средний (зависит от L1) | пара часов |
| Fuse stats в demosaic | ~15–18 мс | Средний (большой рефактор) | день |

Шейдерные микрооптимизации выбиты из списка как source регрессий. Двигаться
**от риска к эффекту**, не наоборот.

## Путь 1 (primary, ~11 мс среднее): temporal subsample

3A-алгоритмы обычно обновляются на 10–15 Hz. При 22 Hz кадрах это значит,
что каждый второй stats-кадр 3A просто отбрасывает. **Значит, можно не
считать stats на каждом втором кадре**.

### Реализация

- Host-логика (`VulkanIspPipeline::recordGrallocBlit` или уровнем выше):
  пропускать stats-dispatch на нечётных кадрах.
- При пропуске — stats-буфер остаётся с прошлыми значениями. 3A
  потребитель уже работает с prev-frame stats (есть latency в async
  пайплайне), одно пропущенное обновление — незаметно.
- Опциональный counter `kStatsInterval = 2` как vendor-property для гибкости.

### Что получаем

- Эффективная стоимость stats падает с 22 мс до ~11 мс в среднем.
- Ноль изменений в шейдерах.
- Ноль риска регрессии на текущем пайплайне.
- Возможность довести `kStatsInterval` до 3 при высоком FPS-таргете.

### Ограничения

- При быстрой смене сцены (быстрый pan, резкое изменение освещения) 3A
  реагирует медленнее. Для preview-use-case приемлемо; для burst/video
  может потребовать interval=1.
- Frame pacing: GPU-время становится неравномерным (stats-кадры тяжелее).
  Нужно проверить, что async pipeline не упирается в пик-время.

## Путь 2 (fallback, ~3–5 мс): luma scratch image

Если Путь 1 применён и stats всё ещё узкое горлышко — попробовать luma
scratch.

### Идея

Демозаик уже имеет R, G, B на выходе. Одновременно со scratch-RGBA
записать **1-байтовый luma в отдельный R8 image**. Stats читает luma из R8
вместо 9 раз делать `dot(rgb, coef)`.

### Diff

Demosaic (добавить):
```glsl
layout(r8, binding = 4) writeonly uniform image2D outLuma;

// в конце main:
float Y = (float(R) * 0.299 + float(G) * 0.587 + float(B) * 0.114) / 255.0;
imageStore(outLuma, ivec2(int(x), int(y)), vec4(Y, 0, 0, 0));
```

Stats (поменять lumaAt):
```glsl
layout(set = 0, binding = 2) uniform sampler2D lumaTex;

float lumaAt(int x, int y) {
    return texelFetch(lumaTex, ivec2(clamp(x,0,pc.imgW-1),
                                     clamp(y,0,pc.imgH-1)), 0).r;
}
```

### Почему ожидание снижено (с ~5–7 до ~3–5 мс)

Первая оценка опиралась на 4× bandwidth-reduction (R8 vs RGBA8). На Tegra
K1 L1 line скорее всего 128 байт независимо от texel-size → "4×" не
реализуется напрямую, выигрыш будет по другой оси: **один dot-product в
demosaic вместо 9 в stats, плюс меньший footprint L1 за счёт денсер
packing**. Конкретика зависит от того, насколько реально sampler2D на R8
даёт меньший footprint в Kepler L1.

**Обязательно замерять на железе перед коммитом.** Если не даёт выигрыша —
откатить и не тратить время.

### Стоимость

- +2 МБ писать в demosaic (один lin→Y + imageStore).
- +2 МБ scratch-image в памяти (R8 1920×1080).
- +1 barrier между demosaic и stats.

## Путь 3 (final, ~15–18 мс): fuse stats в demosaic

Правильное архитектурное решение — не делать stats отдельным проходом.
Большая перестройка, но самый большой выигрыш и единственный способ
гарантированно выкатить 30+ fps.

### Идея

Demosaic WG = 16×16 выходных пикселей. В 1080p это 120×68 WG = 8160 WG.
Каждая WG:

1. Пишет RGBA в scratch (как сейчас).
2. Рассчитывает `Y = 0.299R + 0.587G + 0.114B` в регистре.
3. Кладёт luma в `shared float wgLuma[16][16]`.
4. `barrier()`.
5. Потоки на внутренней 14×14 считают Sobel из shared (8 shared-reads +
   FMA, 1 if-check на край WG).
6. Аккумулирует per-WG local histogram, `rgbSumWG`, `sharpSumWG`.
7. По `gl_WorkGroupID` (константа на WG) определяет главный патч и
   атомически добавляет в per-patch буфер.

### WG ↔ патч выравнивание

WG 16×16 не выровнен по 16×16 патч-сетке патчей в пикселях (≈120×68, WG
16×16 пикселей). Один WG может пересекать границу двух патчей.

**Простой вариант:** считать всю WG в патч, в который попадает её центр
(определяется по `gl_WorkGroupID`, константа на WG). Погрешность на
границах патчей незначительна для AE/AF/AWB.

**Честный вариант:** каждый поток смотрит на свой `patchIdx` и атомически
аккумулирует. Больше атомиков, точнее.

### Что получаем

- 8160 atomicAdd на pass vs текущие 32K → меньше contention.
- Нет отдельного stats-диспатча → 18M `texelFetch` + 18M `dot` из stats
  полностью исчезают.
- Reducer для гистограммы (как во второй оценке первого ревью) всё ещё
  нужен либо перекрывается лёгким атомическим шаблоном.

### Стоимость

- Заметный рефакторинг demosaic + удаление StatsCompute.
- Перекомпоновка host-логики диспатча.
- Один barrier внутри demosaic WG для luma-shared.

### Когда идти на fusion

Если temporal subsample + luma scratch не дают 30+ fps, или нужна полная
производительность без компромиссов по частоте 3A.

## Про NEON — мой взгляд

NEON имеет смысл только если:

1. Готовы полностью отвязать stats от GPU-pipeline'а — считать на CPU из
   **Bayer-буфера** параллельно с GPU-демозаиком, не из scratch (чтобы не
   вводить GPU→CPU sync-point).
2. На CPU достаточно бюджета (Tegra K1 — 4×Cortex-A15 @ ~2.3 ГГц).

### Плюсы

- Stats полностью исчезает из frame time GPU.
- CPU-нагрузка распределяется на 4 ядра.

### Минусы

- Статистика считается по сырому Bayer, без WB/CCM. Меняется семантика
  lumaHist и rgbMean. 3A придётся переписать для нового пространства.
- Два параллельных пайплайна сложнее дебажить.
- DRAM-трафик Bayer ~3 ГБ/с на 22 FPS съедает CPU-bandwidth.

### Вывод

NEON — последний рубеж. Temporal subsample решает половину проблемы за 30
минут и 0 риска. Fusion решает задачу целиком одним тяжёлым рефактором.
NEON добавляет новую ось сложности (семантический mismatch с картинкой) и
не даёт явного преимущества.

## Краткий урок для будущих ревью

**Не доверять арифметической модели без измерения на Tegra K1.** Kepler
SM 3.2 в Tegra K1 — урезанный вариант desktop-Kepler:

- Единственный SMX (192 cores).
- Маленький L1 tex cache (≤12 КБ).
- Старый Vulkan 1.0 драйвер.
- При высоком GPU-load возможен thermal throttle.

Оптимизации, которые выглядят выигрышными в общей модели (warp-coalescing,
divmod-elimination), могут регрессировать из-за L1 footprint или skew.
**Измерять каждый коммит. Откатывать регрессии немедленно.**
